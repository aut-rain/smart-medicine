# 错误码规范说明

## 错误码体系

本项目采用统一的错误码枚举`ErrorCodeEnum`,所有业务异常都应该使用标准错误码。

### 错误码分类

| 分类 | 错误码范围 | 说明 |
|------|-----------|------|
| 成功 | 200 | 操作成功 |
| 系统错误 | 500 | 系统内部错误 |
| 通用错误 | 1xxx | 系统通用错误、参数校验等 |
| 认证授权 | 2xxx | 登录、注册、Token相关 |
| 用户相关 | 3xxx | 用户信息管理 |
| 疾病相关 | 4xxx | 疾病查询等 |
| 药品相关 | 5xxx | 药品查询等 |
| 反馈相关 | 6xxx | 用户反馈 |
| 历史记录 | 7xxx | 浏览历史 |
| 文件相关 | 8xxx | 文件上传下载 |
| AI聊天 | 9xxx | AI对话相关 |

## 使用方式

### 1. Controller层返回错误

```java
// 方式1: 使用错误码枚举(推荐)
if (userId == null) {
    return Result.fail(ErrorCodeEnum.UNAUTHORIZED);
}

// 方式2: 使用错误码枚举+自定义消息
return Result.fail(ErrorCodeEnum.USER_NOT_FOUND, "用户ID:" + userId + "不存在");
```

### 2. Service层抛出异常

```java
// 方式1: 使用错误码枚举(推荐)
throw new BusinessException(ErrorCodeEnum.USER_NOT_FOUND);

// 方式2: 使用错误码枚举+自定义消息
throw new BusinessException(ErrorCodeEnum.PASSWORD_ERROR, "密码错误,还剩" + remainTimes + "次机会");

// 方式3: 使用错误码枚举+原始异常
try {
    // 业务逻辑
} catch (Exception e) {
    throw new BusinessException(ErrorCodeEnum.SYSTEM_ERROR, e);
}

// 兼容旧版本(不推荐)
throw new BusinessException("用户不存在");
```

### 3. 响应示例

#### 成功响应
```json
{
  "code": "200",
  "message": "操作成功",
  "data": {...},
  "timestamp": 1638345600000
}
```

#### 失败响应
```json
{
  "code": "2001",
  "message": "未登录或登录已过期,请先登录",
  "data": null,
  "timestamp": 1638345600000
}
```

## 添加新错误码

在`ErrorCodeEnum`中添加新的错误码：

```java
/**
 * 示例错误
 */
EXAMPLE_ERROR("6002", "这是示例错误描述"),
```

## 注意事项

1. **优先使用错误码枚举**: 所有新代码必须使用`ErrorCodeEnum`,避免硬编码错误消息
2. **错误码不可修改**: 一旦定义错误码,不要随意修改code值,会影响前端判断逻辑
3. **消息可以自定义**: 可以使用`Result.fail(ErrorCodeEnum, customMessage)`提供更详细的错误信息
4. **前后端约定**: 前端根据code进行业务判断,不要依赖message字段
5. **日志记录**: Service层抛出异常会被全局异常处理器捕获并记录日志
