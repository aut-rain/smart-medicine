## 🏗️ 业务模块划分

### 核心业务模块

| 模块 | 包名 | 职责 | 核心组件 |
|------|------|------|----------|
| **认证模块** | `auth` | 用户注册、登录、登出、Token管理、邮箱验证码 | AuthController, AuthService |
| **用户模块** | `user` | 用户信息管理、个人资料、密码修改 | UserController, UserService |
| **疾病模块** | `illness` | 疾病信息管理、查询、热门疾病 | IllnessController, IllnessService |
| **药品模块** | `medicine` | 药品信息管理、查询、疾病关联药品 | MedicineController, MedicineService |
| **AI模块** | `ai` | 智能问答、流式响应、会话记忆管理 | AiChatController, AiChatService, RedisChatMemory |
| **反馈模块** | `feedback` | 用户反馈收集、管理员处理 | FeedbackController, FeedbackService |
| **历史模块** | `history` | 浏览历史记录、浏览量统计 | HistoryController, HistoryService |
| **文件模块** | `file` | OSS文件上传下载 | FileService (REST服务已迁移至其他模块) |

### 共享模块

| 模块 | 包名 | 职责 | 说明 |
|------|------|------|------|
| **数据模型** | `model` | 实体类(Entity)、共享VO | 被所有业务模块使用 |
| **数据访问** | `mapper` | MyBatis数据访问接口 | 被所有业务模块使用 |
| **公共基础** | `common` | 常量、枚举、异常、响应、工具、客户端 | 所有层都可使用 |
| **配置类** | `config` | Spring安全、Redis、AI、CORS、Swagger | 全局配置 |

## 📊 依赖关系

### 依赖方向(从上到下)
```
api (Controller)
  ↓ 依赖
application (Service)
  ↓ 依赖
domain (Entity/DTO/VO)
  ↓ 依赖
infrastructure (Mapper/Cache/File)
  ↓ 依赖
common (Constant/Enum/Exception)
```

### 依赖规则
1. ✅ **上层可以依赖下层**: Controller → Service → Infrastructure
2. ❌ **下层不能依赖上层**: Service不能依赖Controller
3. ✅ **同层之间可以依赖**: Service之间可以相互调用
4. ✅ **所有层都可以依赖common**: 公共基础被所有层使用
5. ❌ **避免循环依赖**: A依赖B,B不能依赖A

## 🎯 Service接口与实现分离的优势

### 为什么要接口与实现分离?

1. **便于切换实现**
   ```java
   // 可以有多种实现
   public class UserServiceImpl implements UserService { }
   public class UserServiceCachedImpl implements UserService { }
   ```

2. **便于Mock测试**
   ```java
   @Test
   void testController() {
       UserService mockService = mock(UserService.class);
       // 测试Controller时不需要真实的Service实现
   }
   ```

3. **面向接口编程**
   ```java
   // Controller依赖接口,不依赖具体实现
   private final UserService userService;
   ```

4. **清晰的职责划分**
   - 接口: 定义做什么(What)
   - 实现: 定义怎么做(How)

## 🔄 数据流转示例

### 用户注册流程

```
1. 前端提交注册表单
   ↓
2. Controller接收请求
   @PostMapping("/register")
   public Result<TokenVO> register(@Valid @RequestBody RegisterDTO dto)
   ↓
3. 参数校验(Jakarta Validation自动完成)
   @NotBlank, @Email等注解
   ↓
4. 调用Service
   authService.register(dto)
   ↓
5. Service处理业务逻辑
   - 检查账号是否存在(调用UserMapper)
   - 密码加密(使用PasswordEncoder)
   - 保存用户(调用UserMapper.insert)
   - 生成Token(调用JwtTokenProvider)
   ↓
6. 返回VO
   return TokenVO(token, refreshToken, userInfo)
   ↓
7. Controller封装Result
   return Result.success(tokenVO)
   ↓
8. 前端接收响应
   { code: "200", message: "注册成功", data: {...} }
```

### 1. 缓存策略
```java

public class IllnessServiceImpl implements IllnessService {

    public IllnessVO getById(Integer id) {
        // 首次查询走数据库,后续走缓存
    }
    
    @CacheEvict(value = "illness", key = "#id")
    public void updateIllness(Integer id, IllnessDTO dto) {
        // 更新时清除缓存
    }
}
```

### 2. 分页查询
```java
public PageResult<IllnessVO> listIllnesses(PageQuery query) {
    Page<Illness> page = new Page<>(query.getPage(), query.getSize());
    Page<Illness> result = illnessMapper.selectPage(page, null);
    return PageResult.of(result, this::convertToVO);
}
```

### 3. 批量操作
```java
public void batchSave(List<IllnessDTO> dtoList) {
    List<Illness> entities = dtoList.stream()
        .map(this::convertToEntity)
        .collect(Collectors.toList());
    
    // 使用MyBatis Plus批量插入
    saveBatch(entities);
}
```

## 🔐 安全设计

### 1. JWT认证流程
```
1. 用户登录 → 生成Token
2. 前端存储Token → LocalStorage
3. 后续请求携带Token → Header: Authorization: Bearer {token}
4. JwtAuthenticationFilter拦截 → 验证Token
5. 解析用户信息 → 存入SecurityContext
6. Controller获取当前用户 → @AuthenticationPrincipal
```

### 2. 权限控制
```java
@PreAuthorize("hasRole('ADMIN')")
public void deleteUser(Integer id) {
    // 只有管理员可以删除用户
}

@PreAuthorize("@userSecurity.isSelfOrAdmin(#id)")
public UserVO updateUser(Integer id, UserUpdateDTO dto) {
    // 只能修改自己的信息,或管理员可以修改所有人
}
```

## 📝 总结

### 架构优势
1. ✅ **职责清晰**: 每一层职责单一,易于理解和维护
2. ✅ **低耦合**: 通过接口依赖,而非具体实现
3. ✅ **高内聚**: 相关功能聚合在同一模块
4. ✅ **易测试**: 接口与实现分离,便于Mock和单元测试
5. ✅ **易扩展**: 新增功能只需新增对应层的代码
6. ✅ **符合规范**: 遵循阿里巴巴开发手册和DDD思想

### 开发原则
1. **按需创建**: DTO/VO不是必须的,只在需要时创建
2. **避免冗余**: 不要为了分层而分层
3. **单一职责**: 一个类只做一件事
4. **面向接口**: 依赖接口而非实现
5. **统一规范**: 遵循命名和编码规范
